# 정보처리기사 실기

## Ch01. 요구사항 확인

### 소프트웨어 생명주기 모델

생명주기(Life Cycle) : 소프트웨어가 탄생하는 과정

- 현대 사회에서 소프트웨어는 방대한 코드 베이스가 존재
- 개발자들 간의 협업을 통해 코드를 생성

소프트웨어 생명주기의 여러가지 모델

1. 폭포수 모델
2. V-모델
3. 프로토타이핑 모델
4. 스파이럴 모델
5. 애자일 모델

#### 폭포수 모델

폭포수 모델(Waterfall Model) : 소프트웨어 개발 과정이 일련의 단계를 통해 순차적으로 진행되는 방식의 모델

폭포수 모델의 단계(요설구테유)

1. 요구사항 분석
2. 설계
3. 구현
4. 테스트
5. 유지보수

폭포수 모델의 특징

1. 각 단계가 완료된 후에 다음 단계로 넘어간다

- 이전 단계로 돌아갈 수 없다

2. 단계가 명확하게 구분된다
3. 문서화가 철저하다
4. 큰 프로젝트에 적합하다
5. 유연성이 부족하다

#### V-모델

V-모델(V-Model) : 폭포수 모델을 변형한 모델로, 개발 과정의 각 단계에 대해 대응되는 테스트 단계가 존재하는 모델

- V-모양의 구조를 통해 각 단계가 명확히 대응됨

V-모델의 단계

1. 요구사항 분석
2. 시스템 테스트, 시스템 설계
3. 통합 테스트, 상세 설계
4. 단위 테스트, 구현

V-모델의 특징

1. 각 개발 단계마다 테스트를 병행

- 결함을 조기에 발견

2. 초기 단계에서 요구사항이 명확한 경우에 적합

#### 프로토타이핑 모델

프로토타이핑 모델(Prototyping Model) : 초기 시제품(프로토타입)을 만들어 사용자의 피드백을 받아가며 점진적으로 개선하는 방식

프로토타이핑 모델의 단계(요설구평수개)

1. 요구사항 수집
2. 빠른 설계
3. 프로토타입 구축
4. 사용자 평가
5. 프로토타입 수정
6. 최종 시스템 개발

프로토타이핑 모델의 특징

1. 사용자의 요구사항을 명확히 파악 가능하다
2. 사용자의 피드백을 반영하여 시스템을 개선할 수 있다
3. 불확실성이 높은 프로젝트에 유용하다

- 스타트업에서 많이 사용

#### 스파이럴 모델

스파이럴 모델(Spiral Model) : 위험 분석을 중심으로 반복적인 개발 과정(Iteration)을 통해 점진적으로 완성도를 높이는 모델링

- 점진적 모델이라고도 표현
- 각 반복(이터레이션) 마다 목표를 설정하고 위험 분석, 개발, 평가를 진행

스파이럴 모델의 단계

1. 목표 설정
2. 위험 분석
3. 개발 및 테스트
4. 평가
5. 다음 목표 설정 등 반복

스파이럴 모델의 특징

1. 위험 관리를 중시한다
2. 반복적인 개발을 통해 유연성과 적응성을 높인다
3. 대규모의 복잡한 프로젝트에 적합하다

#### 애자일 모델

애자일 모델(Agile Model) : 소프트웨어 개발을 유연하고 반복적으로 수행하는 방식

- 고객의 요구사항 변화에 신속히 대응
- 짧은 개발 주기(스프린트)를 통해 기능을 점진적으로 추가

애자일 모델의 단계

1. 요구사항 수집
2. 계획
3. 설계
4. 개발
5. 테스트
6. 리뷰
7. 배포
8. 피드백

애자일 모델의 특징

1. 팀 간의 협업과 고객의 참여를 강조한다
2. 지속적인 개발을 추구한다
3. 작은 팀과 빠른 변화가 요구되는 프로젝트에 적합하다

- 스타트업에서 자주 활용

### 애자일

애자일은 사실 특정 모델이 아니다

- 짧은 주기를 통해 반복적으로 고객의 변화에 신속하게 대응하는 모든 모델링

애자일 방법의 종류

1. 스크럼
2. 칸반
3. XP
4. Lean

#### 스크럼

스크럼(Scrum) : 반복적인 짧은 개발 주기(Sprin)를 통해 점진적으로 소프트웨어를 개발하는 프레임워크

- 팀원 간의 협업과 정기적인 회의를 통해 지속적으로 제품을 개선

스크럼의 특징

1. 스프린트(Sprint) : 2 ~ 4주간의 짧은 개발 주기
2. 일일 스탠드업(매일 짧은 미팅을 통한 진행 상황 공유)
3. 스프린트 리뷰 및 회고 : 스프린트 종료 후 결과물 검토 및 개선점 논의

#### 칸반

칸반(Kanban) : 작업 흐름을 시각화하고, 진행 중인 작업의 수를 제한하여 효율성을 높이는 방법론

- 진행 상태를 시각적으로 관리하여 문제를 신속히 파악하고 해결

칸반의 특징

1. 칸반 보드 : 작업 진행 상태를 나타내는 보드
2. 작업 제한 : 각 단계에서 동시에 진행할 수 있는 작업의 수를 제한
3. 지속적 흐름 : 작업이 완료되면 새로운 작업을 즉시 시작

#### XP

XP(Extreme Programming) : 소프트웨어의 품질을 높이고 변화에 유연하게 대응하기 위해 짧은 개발 주기와 강력한 피드백 루프를 사용하는 방법론

- 피드백, 존중, 의사소통, 용기, 단순성의 5가지 가치를 지님(피존의용단)

XP의 특징

1. TDD(테스트 주도 개발) : 테스트를 먼저 작성하고 코드를 구현
2. 짝 프로그래밍(Pair Programming) : 두 명의 개발자가 한 컴퓨터에서 코드를 작성
3. 지속적 통합(Continuous Integration) : 코드 변경 사항을 자주 통합 및 빌드

#### Lean

린(Lean) : 낭비를 최소화하고 가치 흐름을 최적화하여 효율성을 높이는 방법론

- 제조업에서 유래한 린 원칙을 소프트웨어 개발에 적용

린의 특징

1. 가치 중심 : 고객에게 가치를 제공하는 작업에 집중
2. 낭비 제거 : 불필요한 작업과 프로세스를 제거
3. 지속적 개선 : 지속적으로 프로세스를 개선

### 비용산정 모형 분류

비용산정 모형의 분류

1. 하향식 산정 방법
2. 상향식 산정 방법

#### 하향식 비용 산정

하향식 비용 산정의 종류

1. 델파이 기법 : 전문가의 의견을 종합하여 특정 주제나 문제에 대한 합의된 예측을 도출하는 방법론

#### 상향식 비용 산정

상향식 비용 산정의 종류

1. 코드 라인 수를 통한 계산(LoC) : 원시 코드 라인수의 낙관치, 중간치, 비관치를 측정하여 예측치를 구한 후 비용 한정
2. 맨먼스(Man Month) : 한 사람의 1개월치 업무의 양으로 측정하는 비용 산정법
3. COCOMO : 보헴(Bohem)이 제시한 모형

- 프로그램 규모에 따라 산정
  - 조직형 : 5만 라인
  - 반 분리형 : 30만 라인 이하
  - 임베디드형 : 30만 라인 이상

### 소프트웨어 요구사항

1. 기능적 요구사항 : 시스템이 제공하는 기능 자체, 서비스에 대한 요구사항

- 기능성, 완전성, 일관성 등

2. 비기능적 요구사항 : 시스템 구축의 제한사항에 대한 요구사항

- 보안성, 이식성, 효율성 등

### 디자인 패턴

디자인 패턴(Design Pattern) : 여러가지 소프트웨어 설계 시 사용하기 위해 미리 설계한 패턴

디자인 패턴의 종류

1. 생성 패턴
2. 구조 패턴
3. 행위 패턴

#### 생성 패턴

생성 패턴(Creational Patterns)의 종류

1. 싱글톤 패턴(Singleton Pattern) : 클래스의 인스턴스가 오직 하나만 생성됨을 보장하며, 그 인스턴스에 대한 전역적인 접근점을 제공하는 패턴

- 인스턴스 생성을 제어하여 자원을 효율적으로 관리

2. 팩토리 메서드 패턴(Factora Method Pattern) : 객체를 생성하기 위한 인터페이스를 정의하지만, 인스턴스화할 클래스의 결정을 서브 클래스에서 하도록 하는 패턴

- 객체 생성 코드를 서브클래스에 위임하여 클라이언트 코드의 유연성 증가

3. 추상 팩토리 패턴(Abstract Factory Pattern) : 관련된 객체 군을 생성하기 위한 인터페이스를 제공하며, 구체적인 클래스는 지정하지 않는 패턴

- 관련 객체의 패밀리를 생성하기 위한 인터페이스를 제공
- 객체 생성의 추상화를 통해 제품군의 일관성을 유지

4. 빌더 패턴(Builder Pattern) : 객체 생성을 단계별로 진행하며, 최종적으로 완성된 객체를 반환하는 패턴

- 동일 생성 절차에서 서로 다른 표현 생성 가능
- 복잡한 객체의 구축과 표현을 분리
- 복잡한 객체를 단계적으로 생성하여 코드의 가독성 증가

5. 프로토타입 패턴(Prototype Pattern) : 기존 객체를 복제하여 새로운 객체를 생성하는 방법을 제공하는 패턴

- 복제가 비용 효율적일 경우 유용
- 객체 생성 시 초기 설정을 그대로 유지

#### 구조 패턴

구조 패턴(Structural Patterns)의 종류

1. 어댑터 패턴(Adapter Pattern) : 호환되지 않는 인터페이스를 가진 클래스들을 함께 작동할 수 있도록 변환해주는 패턴

- 기존 코드를 수정 없이 재사용 가능

2. 브릿지 패턴(Bridge Pattern) : 구현부에서 추상층을 분리하여 각자 독립적으로 변형할 수 있도록 하는 패턴

- 추상화와 구현을 분리하여 시스템의 확장성, 유지보수성 증가

3. 컴포짓 패턴(Composite Pattern) : 객체를 트리 구조로 구성하여 부분 - 전체 계층을 표현하는 패턴

- 클라이언트가 개별 객체와 객체 그룹을 동일하게 처리 가능
- 복합 객체와 단일 객체를 동일하게 취급
  - 트리 구조를 쉽게 관리

4. 데코레이터 패턴(Decorator Pattern)
