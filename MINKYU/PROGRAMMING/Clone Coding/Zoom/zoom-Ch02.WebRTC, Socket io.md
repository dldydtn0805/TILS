## Zoom

## Ch01. Zoom Clone Coding

### Protocol

프로토콜(Protocol) : 네트워크 통신에서 데이터를 교환하기 위해 정해진 규칙과 절차의 집합

- 컴퓨터 간 통신을 위해 약속된 언어
- 데이터 송/수신 방식, 데이터 형식, 오류 발생 시 처리 방식 등을 정의
- HTTP, FTP, SMTP등이 모두 프로토콜의 예시이다

### WebSocket

WebSocket : 단일 TCP 연결 상 전이중 통신 채널 제공 프로토콜

- 기존 HTTP 통신 : 클라이언트 요청(Request)에 대해 서버가 응답(Response)하는 방식(단방향)
- WebSocket 방식 : 서버와 클라이언트가 한 번 연결을 맺을 경우, 언제든지 데이터를 서로 전달(양방향)

### WebRTC

WebRTC(Web Real-Time Communication) : 웹 브라우저와 모바일 애플리케이션 간에 별도의 플러그인 없이 실시간으로 음성, 영상 통화 및 데이터 공유를 가능하게 하는 기술들의 집합

- 미디어 및 데이터 스트림을 별도의 서버 없이 브라우저 간에 직접 P2P(Peer-to-Peer) 통신이 가능
- 시그널링 과정이 필요
  - 시그널링(Signaling) : 상대방의 네트워크 정보(IP 주소 등)을 알아내어 연결 방식을 협상하는 과정
  - WebRTC는 시그널링 방법을 정의하지 않음 -> HTTP, WebSocket, Socket.io 등의 다른 통신 기술을 사용하여 시그널링 서버를 구축해야 한다

#### simple-peer

simple-peer : WebRTC의 P2P 통신을 간편하게 구현할 수 있도록 도와주는 라이브러리

- WebRTC의 복잡한 부분을 추상화하여 개발자가 쉽게 P2P를 구현할 수 있도록 도움
- P2P 통신으로 인해 서버를 거치지 않아 지연 시간 감소
- 다양한 브라우저와 플랫폼에서 동작, 서버 인프라 추가 없이 확장 가능

### Socket.io

Socket.io : 실시간 양방향 이벤트 기반 통신을 구현하도록 돕는 JavaScript 라이브러리

- WebSocket 프로토콜을 기본 통신 수단으로 사용
  - WebSocket 연결이 불가능한 경우, 다른 기술로 자동 전환(Fallback)하여 어떤 환경에서든 안정적으로 실시간 통신을 보장
- 단순한 WebSocket보다 높은 수준의 추상화 및 편리한 기능 제공

### 정리

기본적으로 HTTP는 클라이언트(요청) -> 서버(응답)의 단방향 구조이다

이를 해결하기 위해, 양방향 통신이 가능한 WebSocket, Socket.io 등을 활용한다

양방향 통신을 하기 위해, 서로의 네트워크 주소 정보를 알아야 하는데, 이를 알아내는 과정을 시그널링(Signaling)이라 칭한다

WebRTC는 P2P 연결 설정을 위한 시그널링 채널을 구축하기 위해 사용된다

- Socket.io는 안정적인 WebSocket 통신을 지원하기 때문에, WebRTC 시그널링 서버 구현에 편리하다
