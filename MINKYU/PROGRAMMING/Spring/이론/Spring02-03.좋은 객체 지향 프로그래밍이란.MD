### 02-03. 좋은 객체 지향 프로그래밍이란?

### OOP

앞서 설명한 스프링의 탄생 배경 : 좋은 객체 지향 애플리케이션을 개발할 수 있도록 도와주기 위한 프레임워크

그렇다면, 좋은 객체 지향 프로그래밍이 무엇인가??

객체 지향의 특징

1. 추상화
2. 캡슐화
3. 상속
4. 다향성

그렇다면, 이 4가지를 잘 지킨 프로그래밍이 당연히 좋은 객체 지향 프로그래밍이겠구만...

객체 지향 프로그래밍(Object Oriented Programming, OOP) : 컴퓨터 프로그램을 명령어의 목록으로 보는 시각에서 벗어나 여러 개의 독립된 단위(객체)들의 모임으로 파악하려는 것

- 각각의 객체는 메시지(message)를 주고받고, 데이터를 처리할 수 있다
- 프로그램을 유연하고 변경이 용이하게 만든다
  - 컴포넌트를 쉽고 유연하게 변경하(Polymorphism)며 개발 할 수 있는 프로그래밍
  - 대규모 소프트웨어 개발에 많이 사용

#### 다형성의 중요성

다형성의 실세계 비유

- 역할(interface), 구현(object)으로 세상을 구분

ex. 운전자 - 자동차

1. 운전자의 역할이 있고, 자동차의 역할이 있다
2. 운전자의 역할을 활용할 구현이 있고, 자동차의 역할을 활용할 구현이 있다
3. 운전자 구현이 바뀌어도, 역할은 그대로이다. 자동차도 마찬가지이다

- 차종이 바뀌었다고 해서 운전자 입장에서 역할을 그대로 행하면 된다

그 이유는?

- 운전자는 자동차 인터페이스만 알고 있으면, 그 인터페이스를 따서 만든 자동차 구현체를 조종할 수 있기 때문
- 즉, 자동차 역할만 구현할 수 있다면 세상을 바꾸지 않고도 새로운 차종을 만들 수 있다

즉, 클라이언트(운전자)에 영향을 주지 않고도 새로운 기능(차종)을 구현할 수 있다

- 역할과 구현으로 세상을 구분함으로써 이것이 가능하다

역할과 구현으로 세상을 구분하면 단순해지고 유연해지며 변경도 편리해 진다

- 클라이언트는 대상의 역할(interface)만 알면 된다
- 클라이언트는 구현 대상(object)의 내부 구조를 몰라도 된다
- 클라이언트는 구현 대상의 내부 구조가 변경되어도 영향을 받지 않는다
- 클라이언트는 구현 대상 자체를 변경(obj1 -> obj2)해도 영향을 받지 않는다

즉, 자바 언어는 다형성을 활용해서 역할(interface)와 구현(class, object)를 구분한다

- 객체를 설계할 때, 역할과 구현을 명확하게 분리해야 한다
  - 객체 설계 시 역할(interface)를 먼저 부여 -> 해당 역할을 수행하는 구현 객체(object)를 만들기

객체의 협력 : 혼자 있는 객체는 없다

- 클라이언트 : 요청 / 서버 : 응답
- 수 많은 객체 클라이언트와 객체 서버는 서로 협력 관계를 가진다(요청 - 응답)
  - 클라이언트가 서버에게 요청을 보낼 수 있음
  - 클라이언트는 또 다른 서버가 될 수 있음
  - 서버는 해당 요청에 대한 응답을 클라이언트에게 보낼 수 있음

#### 자바 언어의 다형성

Java의 다형성 구현 : Overriding

- Overriding : 재정의
  - 오버라이딩 된 메서드가 실행되는 것
- 다형성으로 인터페이스를 구현한 객체를 실행 시점에 유연하게 변경 가능
- 물론 클래스 상속 관계 또한 다형성과 오버라이딩을 적용 가능

#### 다형성의 본질

다형성을 통해 우리는

1. 인터페이스를 구현한 객체 인스턴스를 실행 시점에 유연하게 변경 가능
2. 객체 사이의 관계(협력)을 통해 다형성의 본질을 이해
3. 클라이언트를 변경하기 않고, 서버 구현 기능을 유연하게 변경 가능

### 다시 보는 스프링 - 스프링과 객체 지향

앞서 설명한 내용

1. 스프링은 좋은 객체 지향 애플리케이션을 개발하기 위해 만들어진 프레임워크이다
2. 좋은 객체 지향 애플리케이션을 만들기 위해 필수적인 핵심 기능은 다형성이다

- 역할과 구현을 구분하여 세상을 바라보는 것

스프링이 좋은 객체 지향 애플리케이션을 만들기 위해 해주는 기능

- 스프링은 다형성을 극대화해서 이용할 수 있게 도와준다
  - 제어 역전(IoC)
  - 의존관계 주입(DI)
- 레고 블럭을 조립하듯이, 차종을 변경하듯이, 공연 무대 배우를 선택하듯이 구현을 편리하게 변경할 수 있다.
